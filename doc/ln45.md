# ln45 — Linker and Library System Design

This document analyzes the requirements for creating a linker (`ln45`) and a static library system for the `cc45` compiler and `ca45` assembler toolchain.

## Current State: Direct-to-Binary
Currently, `ca45` operates as a single-pass absolute assembler.
- **Symbol Resolution**: All symbols (labels, equates) must be known at assembly time.
- **Output**: Generates a raw `.bin` file starting at a fixed `.org` address.
- **Limitation**: Multi-file projects must use `#include` at the source level, forcing the entire project to be re-assembled every time a single file changes.

## The Goal: Relocatable Object Linking
To support modern development workflows and pre-compiled libraries, we need a "Compile -> Assemble -> Link" pipeline.

### 1. Object File Format Specification (`.o45`)
The `.o45` format is a 32-bit extension of the standard Andre Fachat `.o65` relocatable format. It uses the `mode` bit 13 to signal that all header address and length fields are 32-bit.

#### A. Header Structure
| Offset | Field | Size | Value / Description |
|--------|-------|------|---------------------|
| 0      | Marker| 2    | $01, $00 (non-C64)  |
| 2      | Magic | 3    | $6f, $36, $35 ("o65") |
| 5      | Version| 1   | $00 |
| 6      | Mode  | 2    | Bit 13=1 (32-bit), Bit 11=1 (Extended CPU) |
| 8      | Text  | 4    | Load address and length of .text segment |
| 16     | Data  | 4    | Load address and length of .data segment |
| 24     | BSS   | 4    | Base address and length of .bss segment |
| 32     | ZP    | 4    | Base address and length of .zp (Zero Page) |
| 40     | CPU   | 1    | **$45** (New ID for MEGA65 / 45GS02) |

#### B. Relocation Table
Each segment is followed by a relocation table. `.o45` introduces new relocation types to handle the linear 28-bit address space of the MEGA65.

#### C. Symbol Table
Stores both **Exports** (symbols provided by this file) and **Imports** (symbols required from other files).
- Exported symbols include their name and the 32-bit offset within their segment.
- Imported symbols are represented by name; the linker must match these against Exports during the linking phase.

### 2. Relocation Types
The `.o45` format adheres to the standard Andre Fachat `.o65` relocation type values and segment IDs, adding only linear extensions for the MEGA65.

#### A. Standard Segment IDs
- **Segment 2**: Text (Code)
- **Segment 3**: Data (Initialized variables)
- **Segment 4**: BSS (Uninitialized variables)
- **Segment 5**: **Zero Page** (Used for `cc45`'s virtual register pool)

#### B. Relocation Types
| .o45 Type | o65 Base Type | Value (Bits 7-5) | Description |
| :--- | :--- | :--- | :--- |
| **R_M65_WORD** | **WORD** | `0x80` | Standard 16-bit address (e.g., `jsr label`). |
| **R_M65_HIGH** | **HIGH** | `0x40` | High byte of a 16-bit address (e.g., `lda #>label`). |
| **R_M65_LOW**  | **LOW**  | `0x20` | Low byte of a 16-bit address (e.g., `lda #<label`). |
| **R_M65_SEGMENT**| **SEGMENT**| `0xC0` | Standard bank byte (bits 16-23). |
| **R_M65_LINEAR24**| **LINEAR24**| `0x60` (Ext) | **New**: 24-bit linear address (3 bytes). |
| **R_M65_LINEAR32**| **LINEAR32**| `0xA0` (Ext) | **New**: 32-bit linear address/dword (4 bytes). |

#### Why leverage standard Zero Page (Segment 5)?
Standard `.o65` already provides excellent support for Zero Page. 
- **Automatic Offset**: When the assembler emits a relocation for Segment 5, the linker knows this refers to the relocatable Zero Page pool. 
- **Direct Page Support**: Since the 45GS02 can move its Direct Page (via the `B` register), using Segment 5 allows the linker to patch code that works regardless of where the hardware Zero Page is actually mapped at runtime.

#### Linear Extensions (LINEAR24, LINEAR32)
These are the primary extensions required for the MEGA65's 28-bit address space.
- **R_M65_LINEAR24**: Patches all 3 bytes of a "Long" instruction operand (e.g., `LDAL`).
- **R_M65_LINEAR32**: Patches all 4 bytes of a 32-bit linear pointer, ensuring full transparency across the MEGA65's 256MB linear space.

### 3. Linker Logic (`ln45`)
The linker's primary responsibilities are:
1.  **Aggregation**: Read multiple `.o45` files and merge segments of the same type.
2.  **Address Assignment**: Assign a final starting address to each merged segment.
3.  **Symbol Resolution**: Match every "Import" in every object file to a corresponding "Export".
4.  **Relocation**: Iterate through the relocation table and write the calculated absolute addresses into the code buffer.
5.  **Binary Emission**: Output the final contiguous machine code (optionally as a PRG with a 2-byte header).

## Impact on Existing Tools

### Assembler (`ca45`) Changes
- **New Directives**:
    - `.section <name>`: Switch the current emission target.
    - `.global <symbol>`: Mark a symbol as an Export.
    - `.extern <symbol>`: Declare a symbol that exists in another file.
- **Relocatable Mode**: A flag (`-c`) to produce `.o45` instead of `.bin`. In this mode, `ca45` does not require a `.org` directive.
- **Header Generation**: Logic to emit the extended `.o65` header with the new $45 CPU ID.

### Compiler (`cc45`) Changes
- **Standard Includes**: Instead of emitting `#include "stdio.c"`, the compiler will emit `.extern printf` and let the linker find the implementation.
- **Header Files**: Headers will now only contain declarations (`.extern`) rather than implementations.

## Static Libraries (`ar45`)
The `ar45` archiver will manage `.lib` files:
- **Archive Format**: A simple container format bundling multiple `.o45` files with a global index.
- **Selective Linking**: `ln45` will scan libraries and only pull in object files that resolve currently undefined symbols.

## Next Steps
1.  **Specification**: Finalize the binary layout of the `.o45` format (extending `.o65`).
2.  **ca45 Update**: Implement `.section`, `.global`, and `.extern` support.
3.  **ln45 Prototype**: Create a linker that can combine two simple assembly files and apply relocations.
4.  **cc45 Integration**: Update the C compiler to leverage the new multi-file workflow.
